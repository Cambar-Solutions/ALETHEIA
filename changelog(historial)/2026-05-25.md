# Sesión de trabajo — 25 de mayo de 2026

## Contexto del proyecto

Sistema **CLM (Contract Lifecycle Management)** — gestor del ciclo de vida contractual inspirado en Legalario. Gestiona todo desde que un Solicitante crea una solicitud hasta que un Firmante/Apoderado la suscribe, pasando por revisiones del Administrador, el Abogado y el Aprobador.

**Stack definido:**
- Backend: NestJS (monolito modular) + PostgreSQL + Prisma ORM
- Frontend: Next.js como App Shell + microfrontends vía Module Federation + Redux Toolkit + RTK Query
- Monorepo: pnpm workspaces + Turborepo
- Documentación de API: Swagger (`@nestjs/swagger`)
- Linting/formato: Biome + Commitlint + Husky

**5 roles con privilegios dinámicos:** Solicitante · Administrador · Abogado · Aprobador · Firmante

**7 microfrontends:** `solicitudes-mf` · `contratos-mf` · `documentos-mf` · `flujo-mf` · `firmas-mf` · `reportes-mf` · `admin-mf`

## Puntos importantes a considerar al retomar

- **El documento principal es `ImplementacionReferencia.md`** — es la fuente de verdad de decisiones de arquitectura. Si algo no está ahí, no está decidido.
- **`BaseDatos.md` tiene el schema Prisma completo** con notas de diseño. Antes de tocar el schema, revisar ese archivo para no perder las decisiones ya tomadas (especialmente `RefreshToken` y los campos extra de `Contract`).
- **`HistoriasDeUsuario.md` es el backlog base** — 26 HUs con criterios de aceptación. Cualquier funcionalidad que no esté ahí no forma parte del alcance acordado.
- **La biblioteca de UI (componentes) aún no está decidida** — no asumir MUI ni ninguna otra hasta que se tome la decisión.
- **El despliegue será en AWS** pero aún no se ha definido la estrategia exacta — no documentar nada de infra de producción todavía.
- **RTK Query es la decisión para comunicación frontend → backend** — no Axios, no fetch directo. Usa `fetchBaseQuery` con el token JWT en los headers.
- **Los microfrontends se separan por dominio, no por rol** — todos los roles pueden usar el mismo MF; la separación de lo que ven se hace con `<PrivilegeGuard>` dentro del MF.
- **La numeración de secciones en `ImplementacionReferencia.md` terminó en 19** — si se agrega algo nuevo, continuar desde la 20.
