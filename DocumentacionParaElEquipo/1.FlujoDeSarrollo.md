# Flujo de Desarrollo — Backend CLM

**Stack:** NestJS (monolito modular) · PostgreSQL 16 · Prisma ORM  
**Criterio de orden:** del módulo más independiente al más dependiente según relaciones del schema Prisma y flujo de negocio.

---

## Mapa de Dependencias

```
[Infra / Prisma]
       │
       ▼
[Catálogos: areas · societies · apoderados]
       │
       ▼
[Auth · Users]
       │
       ▼
[catalogs/templates] ─── depende de Society
       │
       ▼
[Contracts]  ←── depende de User, Area, Society
       │
       ├──────────────────────────────┐
       ▼                              ▼
[Workflow · Documents]         [Notifications]
       │                              ▲
       │                              │ (dispara Observer)
       └──────── workflow transitions ┘
       │
       ▼
[Signatures]  ←── requiere contrato en estado SIGNING
       │
       ▼
[Reports]  ←── lee AuditLog + Contract + Document + todos los módulos
```

## Fase 0 — Infraestructura Base

> Sin esto nada puede ejecutarse. No hay lógica de negocio aquí, solo la plataforma.

| # | Qué hacer | Por qué primero |
|---|---|---|
| 0.1 | Monorepo: `pnpm workspaces` + `turbo.json` + `biome.json` + Husky + Commitlint | Base de todo el proyecto. Sin esto no hay CI, no hay linting, no hay estructura. |
| 0.2 | `docker-compose.dev.yml` con PostgreSQL 16 | La BD debe estar corriendo antes de poder hacer cualquier migración. |
| 0.3 | `apps/backend` bootstrapeado: NestJS bare-minimum + `AppModule` + `main.ts` + Swagger + `GlobalExceptionFilter` | El servidor debe arrancar. Swagger y el filtro global se registran una sola vez aquí; si se hacen después quedan fuera de algún módulo. |
| 0.4 | `PrismaService` + `schema.prisma` completo + primera migración | El schema **completo** en un solo commit. Prisma genera el cliente tipado que todos los módulos siguientes importan. Hacer migraciones parciales genera regeneraciones del cliente en cada fase. |
| 0.5 | `packages/shared-schemas` vacío pero configurado | Crear el paquete compartido antes de que los módulos empiecen a exportar tipos, para no mover tipos después. |

**Entregable de fase:** `npm run start:dev` levanta en `localhost:3000`, Swagger en `/api/docs`, BD conectada.

## Fase 1 — Catálogos Base

> `Area`, `Society` y `Apoderado` son referenciados por `User`, `Contract`, `Template` y `Signature`. Deben existir primero para poder hacer seeders de prueba y que las FKs funcionen.

| # | Módulo | Depende de | Endpoints |
|---|---|---|---|
| 1.1 | `catalogs/areas` | Solo Prisma | `GET/POST/PATCH /areas` |
| 1.2 | `catalogs/societies` | Solo Prisma | `GET/POST/PATCH /societies` |
| 1.3 | `catalogs/apoderados` | Solo Prisma | `GET/POST/PATCH /apoderados` |

**Por qué antes que Auth:** `User` tiene `areaId` FK → si no existe la tabla `Area` con datos al crear el primer usuario, las pruebas de auth fallan. El seed inicial (`Area`, `Society`, roles y privilegios) vive aquí.

**Entregable de fase:** CRUD de los tres catálogos funcionando. Seed con al menos un `Area` y un `Society`.

---

## Fase 2 — Auth + Users

> El sistema completo está protegido por JWT + `PrivilegeGuard`. Todos los módulos siguientes requieren que este mecanismo esté activo.

| # | Módulo | Depende de | Detalle |
|---|---|---|---|
| 2.1 | `auth` | Prisma, `Role`, `Privilege`, `UserRole`, `RefreshToken` | Login, refresh, logout. `JwtStrategy`, `JwtAuthGuard`, `PrivilegeGuard`, decoradores `@RequirePrivilege` y `@CurrentUser`. |
| 2.2 | `users` | `auth` (guards), `Area` | CRUD de usuarios, asignación de roles, requiere privilegio `USERS_MANAGE`. |

**Seed necesario aquí:** insertar las 5 filas en `Role` (`SOLICITANTE`, `ADMINISTRADOR`, `ABOGADO`, `APROBADOR`, `FIRMANTE`), las 19 filas en `Privilege`, y las asignaciones en `RolePrivilege` (ver tabla de privilegios en `1-Implementacion.md` §7).

**Entregable de fase:** `POST /auth/login` devuelve `{ accessToken, refreshToken, privileges[] }`. Todos los endpoints posteriores pueden validarse con el guard.

---

## Fase 3 — Templates

> `Template` depende de `Society` (FK opcional) y no es requerida por `Contract` en el flujo principal, pero el Administrador la usa para pre-llenar contratos. Se construye aquí para no bloquear nada y porque tiene cero dependencias de negocio más allá de catálogos.

| # | Módulo | Depende de | Endpoints |
|---|---|---|---|
| 3.1 | `catalogs/templates` | `Society`, `auth` (guards) | `GET/POST/PATCH /templates` — requiere `TEMPLATES_MANAGE` |

**Entregable de fase:** CRUD de plantillas con contenido HTML, filtrables por `societyId`.

---

## Fase 4 — Contracts

> Entidad central del sistema. Todos los módulos de negocio restantes son "hijos" de `Contract`. Se construye el núcleo del contrato **sin workflow todavía**: creación, edición, envío, cancelación, recuperación.

| # | Qué construir | Depende de |
|---|---|---|
| 4.1 | `contracts` module + service + controller + repository | `User` (auth context), `Area`, `Society` |
| 4.2 | `IContractRepository` + implementación Prisma | Patrón Repository — `ContractsService` no toca Prisma directamente |
| 4.3 | DTOs: `CreateContractDto`, `UpdateContractDto`, `SubmitContractDto` | — |
| 4.4 | `AuditLog` como side-effect en `ContractsService` | `AuditLog` tabla ya existe en schema; se inserta en cada mutación del contrato |
| 4.5 | Folio auto-generado | Lógica de negocio interna al service |

**Estado cubierto:** `DRAFT → SUBMITTED` (solo submit). Las transiciones a `ADMIN_REVIEW` en adelante las maneja `WorkflowModule`.

**Entregable de fase:** `POST /contracts`, `GET /contracts`, `GET /contracts/:id`, `PATCH /contracts/:id`, `POST /contracts/:id/submit`, `POST /contracts/:id/cancel`, `POST /contracts/:id/recover`.

---

## Fase 5 — Workflow + Documents + Notifications

> Estos tres módulos se desarrollan en paralelo (por integrantes distintos si aplica) porque todos dependen de `Contract` pero no entre sí, **salvo** que `WorkflowService` dispara `NotificationsService` (patrón Observer). Se integran al final de la fase.

### 5.1 — Workflow

| Qué construir | Detalle |
|---|---|
| `WorkflowStage` CRUD | Config de etapas por el Administrador (`WORKFLOW_CONFIG`) |
| `ContractStateMachine` | Motor de transiciones. Valida que la acción sea válida en el estado actual. |
| `ContractWorkflow` + `WorkflowTransition` | Crea el workflow al primer `SUBMITTED`. Registra cada transición con `performedBy` y `comment`. |
| Cálculo SLA | `horasTranscurridas = now() - contractWorkflow.enteredAt`. Devuelve semáforo en `GET /workflow/:contractId`. |

**Endpoints:** `GET /workflow/:contractId`, `POST /workflow/:contractId/approve|reject|return`, `GET/POST/PATCH /workflow/stages`.

### 5.2 — Documents

| Qué construir | Detalle |
|---|---|
| `DocumentRequirementFactory` | Retorna lista de documentos requeridos según `VendorType` (Física vs Moral). Patrón Factory + principio O. |
| Upload multipart | `POST /documents/:contractId` — guarda `filePath`, `fileSize`, `mimeType`. |
| Versionado | `GET /documents/:id/versions`, `POST /documents/:id/versions`. Incrementa `currentVersion`. |

**Endpoints:** `POST/GET /documents/:contractId`, `GET/POST /documents/:id/versions`.

### 5.3 — Notifications

| Qué construir | Detalle |
|---|---|
| `NotificationsService.notify(userId, title, message, contractId?)` | Método interno que crea un registro en `Notification`. |
| Observer hook en `WorkflowService` | Al avanzar de etapa, llama a `NotificationsService` con el userId del responsable de la siguiente etapa. |
| Endpoints propios | `GET /notifications` (del usuario autenticado), `PATCH /notifications/:id/read`. |

**Orden dentro de la fase:** construir 5.1 y 5.2 en paralelo → integrar Observer de 5.3 en 5.1 al final.

**Entregable de fase:** flujo completo `SUBMITTED → ADMIN_REVIEW → LAWYER_REVIEW → APPROVAL_PENDING → APPROVED` funcionando con documentos y notificaciones.

## Fase 6 — Signatures

> Requiere que `Contract` esté en estado `SIGNING` (lo pone `WorkflowModule` tras `APPROVED`). Depende además de `Apoderado` (catálogo de Fase 1) y del guard de auth.

| Qué construir | Detalle |
|---|---|
| `ISignatureStrategy` | Interface común para `CanvasSignature` y `ElectronicSignature`. Patrón Strategy. |
| `CanvasSignatureStrategy` | Recibe `signatureData` (base64), valida que el contrato esté en `SIGNING`, crea `Signature`. Avanza a `SIGNED`. |
| `ElectronicSignatureStrategy` | Placeholder para fase futura; la interfaz ya existe. |

**Endpoints:** `POST /signatures/:contractId`, `GET /signatures/:contractId`.

**Entregable de fase:** firma registrada → contrato pasa a `SIGNED` → AuditLog actualizado.

---

## Fase 7 — Reports

> Lee de todas las tablas. Es el módulo más dependiente del sistema — no puede ser construido antes porque necesita datos reales generados por los módulos anteriores para ser probado.

| Qué construir | Depende de |
|---|---|
| `GET /reports/contracts?status=&area=&assignee=` | `Contract`, `User`, `Area` |
| `GET /reports/audit/:contractId` | `AuditLog` + `Contract` |
| `GET /reports/export?format=csv` | Todo lo anterior — serializa a CSV |

**Entregable de fase:** reportes filtrables y exportables. Módulo funcional completo.

## Reglas de Transición entre Fases

1. **No avanzar a la fase siguiente hasta que los endpoints de la fase actual pasen pruebas manuales en Swagger** (`/api/docs`).
2. **El schema Prisma se define completo en Fase 0.** Agregar columnas después fuerza regenerar el cliente Prisma en todos los módulos activos.
3. **El seed de roles/privilegios es parte de Fase 2** y debe ejecutarse antes de proteger cualquier endpoint con `@RequirePrivilege`.
4. **`AuditLog` no es un módulo separado** — es un side-effect que se inserta directamente en `ContractsService` y `WorkflowService`. No requiere su propio controlador.
5. **El Observer de Notifications** se inyecta en `WorkflowModule` al completar Fase 5.1 — `NotificationsModule` debe estar construido antes de que `WorkflowModule` haga su integración final.
